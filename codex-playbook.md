# Codex 顶级团队工作法

## 核心原则
- 先定义结果，再开始做事。
- 先读现有上下文，再动手修改。
- 默认用最小改动。
- 一次只解决一个目标。
- 先让模型查找和归纳，再让它执行和修正。
- 复杂任务拆成小步，不把所有背景一次性灌进去。

## 默认工作流
1. 先说清目标。
2. 让 Codex 读取现有文件、配置或文档。
3. 让 Codex 给出最小可行方案。
4. 再执行修改。
5. 最后验证结果。

## 适合顶级团队的提示词结构
```text
目标：
<一句话说明最终要什么>

背景：
<项目、文件、链接、现状>

限制：
- 最小改动
- 不改无关部分
- 先查现有内容
- 需要备份时先备份

交付：
- 直接给结果
- 列出改动文件
- 说明验证方式
```

## 任务拆分法
- 查找：先定位文件、配置、报错、文档入口。
- 归纳：把找到的信息压缩成决策所需的最小集合。
- 修改：只改相关文件。
- 验证：跑检查、看结果、补遗漏。

## 省 token 规则
- 不重复贴长上下文。
- 不把一大堆无关材料一次丢给模型。
- 只保留和当前任务有关的文件与结论。
- 长任务分两轮以上处理。
- 结果能从文件中读到的，就不要再口头重述。

## 何时用轻量档
- 搜索资料
- 整理文档
- 生成模板
- 小范围修改
- 简单问答

## 何时用深度档
- 跨文件排查
- 大范围重构
- 难定位 bug
- 需要权衡多种方案
- 要求高准确性的交付

## 常见团队做法
- 让 AI 先产出草稿，再由人确认。
- 让 AI 先做信息提取，再做行动建议。
- 让 AI 对代码、文档、Git 操作分开处理。
- 对会影响环境的动作保留人工确认。
- 对重复工作使用固定模板。

## 命令说明口径
- 命令名保留原样。
- 解释尽量用中文。
- 如果命令有风险，先说作用和影响。
- 如果命令会修改文件，先说备份方案。

## 可直接复用的日常模板

### 代码任务
```text
目标：
修复这个问题：<问题>

背景：
<文件、报错、复现>

要求：
- 先定位原因
- 最小改动
- 补必要测试
- 不扩大范围

交付：
- 修改了哪些文件
- 怎么验证
- 还有什么风险
```

### 文档任务
```text
目标：
把这批内容整理成一份可直接使用的文档

背景：
<来源、受众、用途>

要求：
- 结构清晰
- 去重
- 适合飞书阅读
- 可直接发布

交付：
- 文档正文
- 简短摘要
```

### 数据/知识库任务
```text
目标：
从知识库中提炼出<主题>的关键信息

背景：
<库名、入口页、目标范围>

要求：
- 先找入口页
- 再下钻子页面
- 输出摘要和缺口

交付：
- 汇总
- 链接
- 未覆盖内容
```

## 运行建议
- 默认轻量，深度任务才升级。
- 默认人工复核关键动作。
- 默认每次只做一个主题。
- 默认把结果沉淀成模板，减少下次重复成本。
