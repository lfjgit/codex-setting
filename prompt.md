# Codex 通用工作提示词

## 目标
把任务说清楚，让 Codex 少猜测、少绕路、少重复，优先直接交付可用结果。

## 适用方式
- 先说目标，再说约束，再说交付格式。
- 一次只做一类任务。
- 能从现有文件、配置、文档里读到的信息，先读再改。
- 需要命令时，优先给能直接执行的命令。

## 推荐模板
```text
目标：
<一句话说明你要什么>

背景：
<相关项目、文件、链接、现状>

要求：
- 优先最小改动
- 不要重写无关部分
- 不清楚的地方先查现有内容
- 如果会影响现有配置，先给备份方案

交付：
- 直接给结果
- 说明改了哪些文件
- 如果有命令，尽量用中文解释命令作用
```

## 常见任务模板

### 1. 改代码
```text
目标：
修复这个问题：<问题描述>

背景：
<文件路径、报错信息、复现方式>

要求：
- 先定位原因
- 最小范围修复
- 补充必要测试
- 不要顺手重构

交付：
- 给出修改后的文件
- 简述修复原因
- 说明验证结果
```

### 2. 整理文档
```text
目标：
把这份内容整理成一份清晰的文档或知识库页面

背景：
<原始内容、来源链接、受众>

要求：
- 结构清晰
- 标题层级明确
- 去掉重复和废话
- 用适合飞书阅读的表达

交付：
- 输出可直接放进飞书的正文
- 如果有必要，附一个目录
```

### 3. 处理飞书资料
```text
目标：
从飞书里整理出<资料主题>的摘要

背景：
<知识库名、文档名、表格名、团队名>

要求：
- 先搜索再读取
- 优先抓总入口和索引页
- 需要时再下钻到子页面
- 把内容压缩成可执行摘要

交付：
- 简短总结
- 关键链接
- 未覆盖的信息缺口
```

### 4. GitHub / PR
```text
目标：
处理这个 PR / 代码问题：<说明>

背景：
<仓库、PR 号、报错或 review>

要求：
- 先看差异再动手
- 只改相关文件
- 不要扩大修改范围
- 需要的话补测试

交付：
- 问题列表
- 修改说明
- 验证结果
```

## 省 token 规则
- 不要把整段聊天历史重复贴一遍。
- 不要把无关截图、日志、长文本一次性全塞进来。
- 能通过文件名、链接、错误码定位的，就不要再口头重复。
- 长任务拆成两步：先定位，再修改。
- 对同一个主题持续工作时，尽量复用上一次结论，不要重新解释背景。

## 更准确的规则
- 先让 Codex 读现有上下文，再给新指令。
- 对“你觉得应该怎么做”的问题，尽量换成“按什么标准判断”。
- 需要恢复时，提前要求备份。
- 如果结果要写回飞书，先说明目标对象是文档、Wiki、Sheet 还是多维表格。

## 推荐默认口径
- 默认先做最小可行方案。
- 默认先查现有资料。
- 默认先给可执行结果。
- 默认不大改结构，除非你明确要求重构。
- 默认命令解释用中文，命令本身保留原样。
